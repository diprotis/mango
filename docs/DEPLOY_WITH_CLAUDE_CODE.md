# Deploy Mango to AWS — runbook for Claude Code

**How to use this:** open a terminal in the repo root and run `claude`, then say:
*"Follow docs/DEPLOY_WITH_CLAUDE_CODE.md to deploy and verify Beta."*
(Claude Code runs on this machine, so it has the shell + AWS credentials this task
needs — unlike the Cowork session that wrote the code, which has no AWS access.)

---

## Goal

Deploy the Mango backend to AWS and **prove it works end to end**: a new user can
authenticate, the catalog is served, and a roadmap is generated by a **real Amazon
Bedrock** call. Target **Beta** first; only do Prod when explicitly asked.

**Definition of done:** `make backend-deploy-verify STAGE=beta PROFILE=diprotis-dev`
exits green — i.e. the deploy succeeds and `tests_integration/live_smoke.py` passes
its authenticated journey (profile → catalog → **real Bedrock roadmap** → library →
reflection → grade → account delete).

## Operating rules (read first)

- **Beta before Prod.** Never deploy `prod` unless the user says so in this session.
- **Stop and ask the user before:** enabling phone/SMS (real spend), enabling a new
  Bedrock model, creating/altering IAM roles outside CDK, or anything that costs
  money or changes account-wide settings.
- **Never commit secrets** (Google/Apple secrets, tokens). They go in Secrets
  Manager via `aws secretsmanager put-secret-value`, never in git.
- **Don't hand-edit** `project.pbxproj` or `cdk.out`. Work through `make`/`cdk`.
- Prefer the `Makefile` targets below over raw commands. Profile defaults to
  `diprotis-dev`; pass `PROFILE=...` to override.
- After each gate, if a command fails, **diagnose and fix using the Troubleshooting
  matrix**, then re-run that gate. Don't proceed past a red gate.

## Step 0 — Preflight

```bash
cd "$(git rev-parse --show-toplevel)"          # repo root
aws sts get-caller-identity --profile diprotis-dev
```

- **Expected:** JSON with an `Account` id.
- **If `InvalidClientTokenId` / `ExpiredToken`:** run `aws sso login --profile diprotis-dev`
  (or configure keys) and retry. Do not continue until this returns an account.

Record the account id and region (`aws configure get region --profile diprotis-dev`,
default `us-east-1`).

## Step 1 — Offline tests (fast fail before touching AWS)

```bash
make backend-test            # unit suite (must be all green)
make backend-e2e-local       # moto-backed full-journey e2e (must pass)
cd backend && python3 -m flake8 . && npx --yes aws-cdk@2 synth -c stage=beta >/dev/null && cd ..
```

- **Expected:** unit + e2e pass, flake8 silent, synth succeeds.
- **If anything fails here, stop** — it's a code problem, not an AWS one. Report it.

## Step 2 — CDK bootstrap (one-time per account/region)

```bash
aws cloudformation describe-stacks --stack-name CDKToolkit --profile diprotis-dev >/dev/null 2>&1 \
  && echo "already bootstrapped" \
  || make backend-bootstrap PROFILE=diprotis-dev
```

## Step 3 — Confirm the Bedrock model (access + id)

The backend calls the model id in `backend/config/beta.json` → `bedrockModelId`
(currently a placeholder: `anthropic.claude-3-5-sonnet-20240620-v1:0`).

```bash
grep bedrockModelId backend/config/beta.json
aws bedrock list-foundation-models --by-provider anthropic --region us-east-1 \
  --query "modelSummaries[].modelId" --output table --profile diprotis-dev
```

- Confirm the configured `bedrockModelId` appears and is the model the user wants.
  **If the user wants a different model** (e.g. a newer Claude), update
  `bedrockModelId` in `backend/config/beta.json` first.
- **Model access** is enabled in the Bedrock console (*Model access* page), which is
  a human action. You can't fully verify access without invoking. Proceed; if the
  roadmap step in Step 5 fails with `AccessDeniedException`, send the user to enable
  model access (see Troubleshooting) and re-run Step 5.

## Step 4 + 5 — Deploy Beta and verify end to end (one command)

```bash
make backend-deploy-verify STAGE=beta PROFILE=diprotis-dev
```

This deploys all stacks, reads the stack outputs from a scoped `--outputs-file`,
then runs `tests_integration/live_smoke.py` (real Cognito + real Bedrock).

- **Expected final line:** `✅ DEPLOY + LIVE SMOKE PASSED (beta)`.
- Capture the printed `API_URL`, `POOL_ID`, `CLIENT` — you'll need them for the app.
- If it fails, use Troubleshooting, fix, and re-run **just the verify** with:
  `cd backend && bash scripts/deploy_and_verify.sh beta diprotis-dev`.

## Step 6 — (Optional) Wire the iOS app to the deployed backend

Only if the user wants to run the app against Beta now. Read the stack outputs and
fill `ios/Mango/Resources/AppConfig.plist`:

```bash
aws cloudformation describe-stacks --profile diprotis-dev \
  --query "Stacks[].Outputs[?OutputKey=='ApiUrl'||OutputKey=='UserPoolClientId'||OutputKey=='UserPoolDomain'].[OutputKey,OutputValue]" \
  --output text
```

Set in `AppConfig.plist` (strip `https://` from the domain for `CognitoDomain`):
`BetaAPIURL` = ApiUrl, `CognitoDomain` = `<host>.auth.us-east-1.amazoncognito.com`,
`CognitoClientId` = UserPoolClientId, `CognitoRegion` = `us-east-1`,
`CognitoRedirectScheme` = `mango`. Then `make ios-test` (or open in Xcode and run).
Email sign-in works immediately; Google/Apple/phone need the steps in
`docs/AUTH_PROVIDERS_SETUP.md`.

## Step 7 — Report back

Summarize for the user: the deployed stage + account/region, the API URL and Cognito
pool/client ids, the smoke result, and any follow-ups (e.g. "Bedrock access needed",
"app plist wired"). Do **not** commit stack outputs or secrets. If you changed
`config/beta.json` (model id) or `AppConfig.plist`, commit those with a clear message.

## Troubleshooting matrix

| Symptom | Cause | Fix |
|---|---|---|
| `InvalidClientTokenId` / `ExpiredToken` | No/expired creds | `aws sso login --profile diprotis-dev` |
| `is not authorized to perform: ... bootstrap` / `CDKToolkit` missing | Not bootstrapped | `make backend-bootstrap PROFILE=diprotis-dev` |
| Deploy: `AccessDenied` creating resources | Deploy identity lacks perms | Use an admin/deploy role; check the profile |
| Smoke roadmap step: `AccessDeniedException` / `model ... not accessible` | Bedrock model access not enabled | Bedrock console → **Model access** → enable the `bedrockModelId` for `us-east-1`; re-run verify |
| Smoke roadmap: `ValidationException` invalid model id | Wrong/region-mismatched model id | Fix `bedrockModelId` in `backend/config/beta.json`; redeploy |
| Smoke authed test fails at `admin_initiate_auth` | App client lacks the admin flow | Already enabled in `auth_stack.py` (`admin_user_password=True`); ensure the **latest** code is deployed |
| `make backend-deploy-verify` aborts "Missing Cognito outputs" | Deploy didn't finish / wrong region | Re-run; confirm same `--profile`/region throughout |
| Hosted-UI domain create fails "already exists" | `cognitoDomainPrefix` collides globally | Leave it blank (account-scoped default) or pick a unique prefix in `config/beta.json` |
| Prod bucket/table won't delete on teardown | Prod uses `RETAIN` (intentional) | Delete the retained resources manually if truly tearing down |

## Rollback / teardown (only if asked)

```bash
cd backend && npx --yes aws-cdk@2 destroy -c stage=beta --profile diprotis-dev --all
```
Prod retains the analytics bucket, feature table, and user pool by design — remove
them by hand only if the user confirms.
