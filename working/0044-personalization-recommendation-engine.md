# 0044 ‚Äî Gen-AI personalization & recommendation engine

- **Epic:** M15 ¬∑ **Status:** Draft ¬∑ **Owner:** unassigned ¬∑ **Updated:** 2026-06-28
- **Reviewers:** Principal/SD/QA

> **Cluster note.** This is the **recommendation brain** of the 0038‚Äì0044 activity cluster. The
> flagship engine [`0038-agentic-roadmap-engine`] owns orchestration and explicitly carves out a
> **Personalizer stage** that "*calls* the 0044 recommender" for "preferred types/modalities, a
> difficulty target" and applies the result as a **thin overlay over the cached base track** (0038
> ¬ß6.2 agent #5, ¬ß6.6, D-4). This spec defines that recommender: how it turns the **0020 feature
> store** + the user's **0026 activity history** + **0015 events** + explicit feedback into a
> ranked, justified set of recommendations ‚Äî *which activity next, which kind/modality mix, which
> difficulty, which content-feed items, when to review (0016), when to nudge (0025), and optionally
> which book next* ‚Äî with **gen-AI agents (Bedrock Claude) reasoning over and re-ranking** a
> candidate set. It honors `docs/GAMIFICATION.md`: it optimizes **learning + healthy engagement**,
> not time-on-app, and ships with transparency + user controls. It does **not** own the feature
> producers ([`0020`]), the activity taxonomy ([`0039`]), the modality runtimes ([`0040`‚Äì`0043`]),
> or orchestration ([`0038`]); it *consumes* them. Forward dependencies that are not yet drafted
> ([`0026`] history, [`0027`] artifact/observability, [`0030`] safety) are treated as **seams** with
> the exact contract pinned where used.

## 1. Summary

Today Mango has **no recommender**. A roadmap is one rigid `4√ó2√ó2` template generated by a single
Bedrock call with no personalization, and the feature store ([`0020`], `MangoFeatures-<stage>`) is
read only to inject a few hints into the generation prompt ‚Äî there is no notion of "what should this
user do *next*", no re-ranking, no learning from feedback. This spec builds the **gen-AI
personalization & recommendation engine**: a **hybrid, three-layer** system ‚Äî **(1) candidate
generation** (content-based from book/activity metadata + collaborative co-occurrence signals from
`MangoFeatures`/events + knowledge-rule candidates from pedagogy and lifecycle state), **(2) a gen-AI
re-ranker agent** (a single Bedrock Claude `invokeModel` call that reasons over the user's feature
vector + recent history + stated goals + explicit feedback to **order** the candidates and pick the
next activity / difficulty / modality, returning a *justification* per pick), and **(3) the
personalization of the track** consumed by 0038's Personalizer (which milestones to emphasize, which
activity kinds/modalities to favor, the difficulty curve). It is served by an online Lambda behind
`GET /v1/me/recommendations` and an internal `recommend(context)` the Step Functions Personalizer
calls; the heavy candidate features are **precomputed** (rides the 0020 batch) and the per-request
cost is one small cached Bedrock call. Every recommendation + the outcome it produced is **logged**
(0015/0027) to close a **feedback loop** feeding **offline evaluation** (NDCG/precision\@k,
diversity, calibration) and, once the **A/B framework gap (G12)** is closed, **online experiments**
with **healthy-engagement guardrail metrics** (reflection/application rate, over-use detection,
notification-disable rate ‚Äî not session count). **Embeddings + vector search are explicitly a v2
layer**, not v1: v1 ranks a metadata + collaborative candidate set with an LLM re-ranker; v2 adds
**Amazon Titan Text Embeddings v2 + Amazon S3 Vectors** for semantic content-feed and next-book
similarity. The outcome: for an active user, the next activity, its difficulty/modality, the daily
review, and the nudge time **demonstrably adapt** to how they actually learn, with a visible "why am
I seeing this" and a one-tap **reset**, and nothing optimizes for compulsion.

## 2. Goals / Non-goals

- **Goals:**
  - A **hybrid recommender** with three clean layers ‚Äî **candidate generation** (content-based +
    collaborative + knowledge-based), a **gen-AI re-ranker agent** (Bedrock Claude), and a
    **track-personalization** output consumed by 0038's Personalizer (¬ß6.1‚Äì¬ß6.4).
  - A precise **signal set**: exactly which 0020 features, 0026 history fields, 0015 events, and
    explicit feedback (thumbs/skip) feed the engine, plus **cold-start defaults** (¬ß6.5).
  - A precise **recommendation surface**: **next activity** within a track, **activity-kind/modality
    mix**, **difficulty target**, **content-feed items** (ties [`0042`]), **review timing** (ties
    [`0016`]), **notification timing** (ties [`0025`]), and optionally **next book** in the catalog
    ([`0009`]) ‚Äî each as a typed, justified recommendation (¬ß6.6).
  - An **online serving** path ‚Äî a Lambda behind `GET /v1/me/recommendations` and an internal
    `recommend(context)` call the 0038 Personalizer uses ‚Äî that reads `MangoFeatures`, runs candidate
    generation, calls Bedrock to re-rank, **caches** the result, and is **float-free** (¬ß6.7‚Äì¬ß6.8).
  - A **feedback loop + evaluation** plan: log `(recommendation, outcome)` to the lake/artifact store,
    an **offline eval harness** (NDCG/precision\@k, diversity/novelty, difficulty calibration), and an
    **online A/B** design gated on the framework gap **G12**, with **healthy-engagement guardrail
    metrics** baked in (¬ß6.9, ¬ß6.10).
  - **Ethical, transparent personalization** per `docs/GAMIFICATION.md`: optimize *learning +
    durable, healthy engagement*; expose **"why am I seeing this"** and a **reset/controls** surface;
    **over-use detection** that gently *de-escalates* rather than maximizing engagement (¬ß6.10).
  - The **data/contract surface**: DDB recommendation + feedback items (float-free), the endpoint +
    openapi, **least-privilege IAM** (Bedrock + `MangoFeatures` read + lake/artifact write), and the
    **privacy** posture (¬ß6.7, ¬ß6.8, ¬ß6.11).
  - Land **incrementally**: ship layers 1+2 (candidate gen + LLM re-rank) serving an identity-safe
    cold-start first, then the feedback loop, then v2 embeddings/vector search (¬ß9).
- **Non-goals:**
  - **The feature producers / feature store** ‚Äî [`0020`] owns `MangoFeatures` population and the
    `shared/features.py` module; this spec is a **consumer** (adds a few features it needs, noted as a
    0020 delta, but does not build the batch/online producers).
  - **The activity taxonomy + grading** ‚Äî [`0039`]; we recommend *over* `typeId`/`modality`/
    `difficulty`/`objectiveRef`, we do not define them.
  - **Orchestration / track assembly / the Verifier** ‚Äî [`0038`]; we return an overlay-shaped
    recommendation the Personalizer applies; we do not run Step Functions or compose the track.
  - **The modality runtimes** ([`0040`‚Äì`0043`]), the **review scheduler** itself ([`0016`]'s
    SM-2-lite is the scheduler; we only bias *which* cards/when surfaces), and the **notification
    delivery** ([`0025`]'s APNs/local plumbing; we only recommend the *hour/which* nudge).
  - **A trained deep model / RL policy / a managed recommender (Amazon Personalize) as the v1
    engine** ‚Äî see ¬ß10 D-2 (build-vs-buy): v1 is candidate-gen + LLM re-rank on serverless primitives
    we already operate; a managed/trained model is a deliberate **later** option once interaction
    volume justifies it.
  - **Real-time streaming personalization**, multi-armed-bandit auto-tuning of XP, or social/league
    ranking ([`0021`]).
  - **Building the A/B experimentation framework** ‚Äî that is the architecture-review gap **G12**;
    this spec *defines what it needs from it* and degrades to offline-only until it lands.

## 3. Background & context

**As-built (verified by reading the code).**
- **No recommender exists.** Generation is `POST /v1/roadmaps/generate` ‚Üí async `roadmap_worker` ‚Üí
  one `agent.generate_roadmap(book, profile, excerpt)` Bedrock call (`backend/src/shared/agent.py`,
  `_invoke(system, user, max_tokens)` ‚Üí `bedrock-runtime:InvokeModel`, IAM auth, adaptive extended
  thinking, `extract_json`) ‚Üí a fixed `4 milestones √ó 2 lessons √ó 2 exercises{quiz|reflection|
  application}` JSON stored as a string on the job row (`shared/roadmap_jobs.py`,
  `PK=USER#<uid>`/`SK=ROADMAPJOB#<jobId>`). There is no "next activity" decision and no learning loop.
- **The feature store is provisioned and (per 0020) about to be populated** but is **not** read by
  any ranking logic: `analytics_stack.py` defines `MangoFeatures-<stage>` (`PK=entityId`,
  `SK=featureName`, PAY_PER_REQUEST, prod PITR); [`0020`] adds `shared/features.py`
  (`get_features`/`put_features`, **float-safe** scaled-int/JSON encoding, cold-start `DEFAULTS`,
  `GET /v1/me/features`) and the producers that fill `reading_pace_wpm`, `pref_topics`,
  `best_hour_local`, `avg_session_min`, `difficulty_tolerance`, `days_active_28`, `lessons_7d`,
  `last_active_ts`. **This spec reads exactly those, plus a small additive set it needs (¬ß6.5).**
- **The events lake exists** ([`0015`]/`0006`): `POST /v1/events` ‚Üí Firehose ‚Üí `s3://‚Ä¶/events/
  dt=‚Ä¶/` (GZIP JSON), Glue `mango_<stage>.events` (`ts,type,userId,props`) queryable in Athena. The
  closed event taxonomy includes `exercise_graded {kind, correct, score, xpAwarded}`,
  `lesson_completed`, `streak_extended/frozen`, `achievement_unlocked` ‚Äî the **outcome signals** the
  feedback loop needs.
- **The 0038 Personalizer is the consumer.** 0038 ¬ß6.2 agent #5 reads "*the assembled base track +
  0020 features + **0044 recommendation** output (preferred types/modalities, difficulty target)*"
  and emits an **overlay** (`{trackId, userSub, selectedActivityIds[], difficultyById{},
  modalityEmphasis, order[]}`, 0038 ¬ß6.6). Cold features ‚áí **identity overlay** (= base). **This
  spec produces exactly what that stage consumes.**
- **Activity vocabulary** ([`0039`]): 11 kinds (`mcq, quiz, puzzle, long_answer, voice,
  conversation, video, image, social_engage, content_consume, peer_session`), `modality ‚àà
  {text, choice, audio, video, image, external_proof, none}`, `difficulty 1..5` Bloom-aligned,
  `gradingMethod`, `objectiveRef`. We recommend *over* these; the **per-type/modality success the
  user has had** (from 0040/0041/0042/0043 graded outcomes) is a core signal.
- **Bedrock seam** (`shared/agent.py`): the re-ranker is a new focused prompt invoked exactly like
  `generate_roadmap`/`grade` ‚Äî `_invoke(system, user, max_tokens)`, `extract_json`, monkeypatchable
  `_runtime` for tests. **IAM only, no API key** (CLAUDE.md). DynamoDB **rejects `float`** ‚Üí scaled
  ints / JSON. Lambdas **stdlib + boto3**. black (100) + flake8 (120).

**Why now (the pivot).** Per [`0008`] Mango delivers **engaging activities** and tracks the journey;
[`0038`]‚Äì[`0043`] make the activity space **rich** (taxonomy √ó modalities √ó difficulty). A rich space
is only valuable if the product **curates** it per learner ‚Äî otherwise every user gets the same
generic track. The recommender is the piece that turns "we *can* generate 11 kinds across 6
modalities at 5 difficulties" into "we serve *you* the right next thing." And `docs/GAMIFICATION.md`
is explicit that the *engagement layer's job is to bridge the delayed-value gap* of education apps
(~14 % D1 / ~2‚Äì3 % D30 retention) **with near-term, intrinsically satisfying, honest rewards** ‚Äî a
recommender that optimizes the wrong objective (raw time-on-app) would violate the product's ethics
manifesto (¬ß6 of that doc). So this engine's objective is **healthy engagement = learning behaviors
sustained over time**, not session count.

**Why gen-AI + hybrid (the research basis).**
- **Hybrid + cold-start.** Pure collaborative filtering fails for new users/items; the established
  fix is a **hybrid** that uses **content-based / knowledge-based** signals to bootstrap until
  collaborative data accrues ([Frontiers hybrid e-learning cold-start](https://www.frontiersin.org/journals/computer-science/articles/10.3389/fcomp.2024.1404391/full),
  [Vinija cold-start notes](https://vinija.ai/recsys/cold-start/)). Mango is **cold-start by default**
  (a brand-new reader, a freshly imported book) so a content/knowledge floor is mandatory.
- **LLM-as-recommender / re-ranking.** The strong, low-cost pattern is **two-stage**: a cheap
  retrieval/candidate stage, then an **LLM re-ranks a small candidate set** (scoring-based ‚Äî score
  candidates / emit index tokens, not free-generate the whole list ‚Äî is the efficient, practical
  mode), giving **zero-shot reasoning + explanations** without training a model
  ([LLM explainable re-ranker, arXiv 2512.03439](https://arxiv.org/pdf/2512.03439),
  [token-controlled LLM re-ranking, arXiv 2511.17913](https://arxiv.org/html/2511.17913),
  [reranking via user-preference retrieval, COLING 2025](https://aclanthology.org/2025.coling-main.45/)).
  The same literature warns LLM rankers are **biased by item popularity and position in the prompt**
  and "struggle to perceive the order of historical interactions" ‚Äî mitigated by careful prompting,
  shuffling, and bootstrapping (handled in ¬ß6.3, ¬ß10 R-2).
- **The barbell finding (why hybrid, not pure-LLM).** 2024 results show **LLM-augmented systems beat
  pure CF on cold-start but *underperform* CF on warm, data-rich pairs** ‚Äî so the research-backed best
  practice is **LLM for feature enrichment + reasoning, collaborative signal for the behavioral
  ground truth** ([build-vs-buy matrix 2026](https://www.digitalapplied.com/blog/ecommerce-recommendation-engines-2026-build-vs-buy-matrix)).
  Our layering reflects this: collaborative co-occurrence shapes the candidate set; the LLM reasons
  and re-ranks; as warm data grows the collaborative layer is weighted up.
- **RAG over history / sequential.** Recommendation can be framed as **next-item prediction** with the
  user's recent interactions as **in-context few-shot demonstrations**; retrieving relevant history
  into the prompt (a light RAG) improves preference capture
  ([generative-rec survey, arXiv 2309.01157](https://arxiv.org/pdf/2309.01157),
  [graph-RAG conversational rec, arXiv 2503.06430](https://arxiv.org/html/2503.06430v1)). v1 uses a
  **compact recent-history window** in the prompt; v2 can retrieve semantically similar past
  successes via embeddings.
- **AWS building blocks.** **Bedrock Claude** does the reasoning/re-rank (already our `agent.py`
  path). For v2 semantic similarity, **Amazon Titan Text Embeddings v2** (256/512/1024-dim) or
  **Cohere Embed** generate vectors, stored cheaply in **Amazon S3 Vectors** (native vector store,
  ~90 % cheaper than cluster stores, 100‚Äì800 ms retrieval) with **Amazon OpenSearch Serverless** as
  the low-latency upgrade if/when needed
  ([S3 Vectors GA](https://aws.amazon.com/blogs/aws/introducing-amazon-s3-vectors-first-cloud-storage-with-native-vector-support-at-scale/),
  [Titan v2 + OpenSearch](https://repost.aws/articles/AR3KH3VFZ0RGWmOTOM5nE8hQ/exploring-embeddings-and-llms-amazon-titan-embedding-v2-claude-3-5-and-opensearch-serverless)).
  The **online feature store** (`MangoFeatures`) serves single-digit-ms features at request time; the
  **S3 lake** is the offline training/eval substrate ‚Äî the textbook online/offline split
  ([Personalize real-time recs](https://aws.amazon.com/blogs/machine-learning/implement-real-time-personalized-recommendations-using-amazon-personalize/),
  [Personalize + Bedrock](https://aws.amazon.com/blogs/machine-learning/generate-user-personalized-communication-with-amazon-personalize-and-amazon-bedrock/)).
- **Evaluation.** Standard practice: **offline** rank metrics (precision/recall\@k, MAP, **NDCG**)
  first, then **online A/B** on real behavior, because offline gains don't always transfer
  ([Shaped offline-vs-online](https://www.shaped.ai/blog/evaluating-recommender-models-offline-vs-online-evaluation)).
  And critically, **"beyond optimizing for clicks"** ‚Äî incorporate **editorial / quality values** as
  explicit objectives so the system doesn't collapse to an engagement-maximizer
  ([beyond-clicks news rec, arXiv 2004.09980](https://arxiv.org/pdf/2004.09980)). For Mango the
  "editorial values" are **learning quality + healthy engagement**.
- **Ethics.** Recommender feedback loops can form **self-reinforcing, addictive engagement loops** and
  exploit vulnerabilities; the antidote is **transparency, user control, natural stopping points, and
  optimizing wellbeing over engagement**
  ([algorithmic addiction by design, arXiv 2505.00054](https://arxiv.org/pdf/2505.00054),
  [design ethics / dark patterns, SilverCloud](https://silvercloud.amwell.com/blog/2022/01/design-ethics-for-mental-health-how-and-why-we-avoid-dark-patterns)).
  This maps directly onto `docs/GAMIFICATION.md` ¬ß6 and is encoded as **guardrails** (¬ß6.10).

**Related specs.** Consumes [`0020`] (features), [`0015`] (events/outcomes), [`0026`] (history),
[`0039`] (activity vocabulary), [`0009`] (catalog, for next-book), [`0027`] (artifact/observability),
[`0030`] (safety). Produces input to [`0038`] (Personalizer overlay), biases [`0016`] (review) and
[`0025`] (nudge timing), ties [`0042`] (content-feed). Pairs with [`0023`] (credits ‚Äî recs are free)
and [`0029`] (rate-limit ‚Äî re-rank is a Bedrock call).

## 4. User stories

- As a **returning learner** who aces quizzes but loves the talk-it-out tutor, my next track **leans
  into conversational/application activities at a higher difficulty** because the engine saw my high
  `difficulty_tolerance` and my strong `conversation`/`long_answer` outcomes and re-ranked toward
  them.
- As a **new learner** (zero history), I still get a **sensible, balanced** first track ‚Äî a gentle
  difficulty, a mix of modalities, topics matching my onboarding interests ‚Äî because the cold-start
  layer (content + knowledge rules) produces a good default and the LLM re-ranks it conservatively.
- As a **learner who skipped three video tasks in a row**, the engine **down-weights `video`** for me
  and surfaces text/audio alternatives ‚Äî explicit negative feedback is respected fast.
- As a **busy user**, on a day I have two minutes, the engine recommends a **short `mcq` or the daily
  Insight Review** ([`0016`]) rather than a 30-minute application task, and the nudge lands at **my
  best hour** ([`0025`]) ‚Äî never more than the GAMIFICATION cap.
- As a **curious learner** who just finished a book, I get a **"what to read next"** recommendation
  from the catalog ([`0009`]) that fits my demonstrated topics and reading pace, with a one-line
  reason.
- As **any user**, I can tap **"Why am I seeing this?"** on a recommendation and get a plain-language
  reason, and I can **reset my personalization** in Settings with one tap; I am never nudged into
  compulsive use ‚Äî if I'm over-using, the app *eases off*.
- As the **0038 Personalizer**, I call `recommend(context)` and get a typed
  `{preferredTypes[], preferredModalities[], difficultyTarget, activityOrderHints[], reasons{}}` I can
  turn into an overlay in one pass; cold features ‚áí I get an **identity-safe** default.
- As a **product owner / safety reviewer**, I can open the **recommendation transcript** (candidates
  in, LLM verdict, final picks, reasons) in the [`0027`] store, and I trust no recommendation bypasses
  the [`0030`] safety posture or the healthy-engagement guardrails.

## 5. Requirements

### Functional

- **FR-1 (hybrid candidate generation).** The engine MUST build a **candidate set** from three
  sources, unioned and de-duplicated, each candidate tagged with its source(s) and a coarse prior
  score: **(a) content-based** ‚Äî candidates whose metadata (`objectiveRef` topic, `typeId`,
  `modality`, `difficulty`, book genre/`pref_topics`) matches the user's profile/features;
  **(b) collaborative** ‚Äî candidates favored by **similar users** via cheap co-occurrence signals
  precomputed from the events lake (e.g. "users with this profile who completed objective X next did
  well on activity-kind Y"); **(c) knowledge-based** ‚Äî pedagogy + lifecycle rules (next *unfinished*
  objective in the scaffold; a `spacedReview` whose card is due; cover an under-practiced Bloom
  level; respect `maxAttempts`/lifecycle from [`0039`]). Candidate generation is **pure/deterministic
  given its inputs** and **fully functional with zero collaborative data** (cold-start = content +
  knowledge only) (¬ß6.2).
- **FR-2 (gen-AI re-ranker agent).** A single **Bedrock Claude** `invokeModel` call MUST **re-rank**
  the candidate set: given the user's **feature vector** (allow-listed 0020 features), a **compact
  recent-history window** (last N graded activities with kind/modality/score), **stated goals/topics**
  (onboarding + `pref_topics`), and **explicit feedback** (recent thumbs/skip), it returns an
  **ordered list of candidate ids with a per-pick reason** and the **chosen next activity, difficulty
  target, and modality emphasis** ‚Äî in **strict JSON** (`extract_json`-parseable). The re-ranker is
  **scoring/ordering over a provided id set** (it MUST NOT invent activities), and the prompt MUST
  apply the **anti-bias mitigations** of ¬ß6.3 (shuffle candidate order, cap set size, instruct against
  popularity bias) (¬ß6.3).
- **FR-3 (track personalization output).** The engine MUST expose a typed result the **0038
  Personalizer** consumes directly: `{preferredTypes[], preferredModalities[], difficultyTarget(1‚Äì5),
  activityOrderHints[] (candidate ids in recommended order), milestoneEmphasis[], reasons{id‚Üístring}}`.
  With **cold features** it MUST return an **identity-safe default** (balanced modalities, difficulty
  from onboarding reading level, no re-order) so the overlay is a no-op (= base track) (¬ß6.4, ¬ß6.6).
- **FR-4 (recommendation surface).** Beyond the next in-track activity, the engine MUST be able to
  recommend: **activity-kind/modality mix** for a track, a **difficulty target**, **content-feed
  items** (`content_consume`/`social_engage` candidates for [`0042`]'s feed), **review timing/which
  cards** (a bias passed to [`0016`]'s scheduler), **notification timing** (`best_hour_local` + a
  "what nudge" choice for [`0025`]), and **next book** from the catalog ([`0009`]). Each is a typed
  `Recommendation{kind, payload, reason, score, ttl}` (¬ß6.6). Surfaces not yet built (feed/review/
  nudge) consume the recommendation via their own seam; absence of a consumer never errors the engine.
- **FR-5 (online serving).** `GET /v1/me/recommendations?surface=<surface>&context=<ctx>` MUST return
  the caller's recommendations for a surface, resolving identity from the JWT (dev `x-mango-user`
  fallback outside prod/beta, per `response.user_id`). An **internal** `recommend(context) -> dict`
  (same module) MUST be callable by the 0038 Personalizer Lambda without HTTP. Both read
  `MangoFeatures` (single-digit-ms), generate candidates, call the Bedrock re-rank, and return within
  the latency budget (NFR-1). The result MUST be **cached** (¬ß6.8) (¬ß6.7).
- **FR-6 (feedback loop).** Every served recommendation MUST be **logged** with a `recId` correlation
  id (the candidates, the LLM verdict + reasons, the final picks, model/tokens/latency) to the
  [`0027`] artifact store, and a compact **`recommendation_served`** + **`recommendation_outcome`**
  event MUST be emitted to the [`0015`] lake (outcome = was the recommended activity started/
  completed, its score, was it skipped/thumbed-down) ‚Äî **best-effort, never failing the response**.
  These power offline eval and (later) A/B (¬ß6.9).
- **FR-7 (explicit feedback intake).** The system MUST accept **explicit feedback** ‚Äî `thumbs_up`/
  `thumbs_down`/`skip`/`too_easy`/`too_hard` on a recommendation or activity ‚Äî via
  `POST /v1/me/recommendations/feedback`, store it (float-free) keyed to the user, and make recent
  feedback available to the re-ranker (FR-2) and the 0020 producers (e.g. nudging
  `difficulty_tolerance`) (¬ß6.6, ¬ß6.7).
- **FR-8 (cold-start defaults).** Defaults MUST live in one place and produce a sensible recommendation
  with **no** features/history: difficulty from onboarding `readingLevel`
  (`casual‚Üí1, focused‚Üí2, deep‚Üí3` as a starting target), a **balanced modality mix** (text + one
  audio/visual taster), topics from onboarding `interests`/`pref_topics`, no collaborative re-rank
  (LLM re-ranks on content+knowledge only). The cold path MUST be **offline-safe** in spirit (no
  collaborative data dependency) and degrade to a pure rules default if Bedrock is unavailable (¬ß6.5).
- **FR-9 (healthy-engagement guardrails).** The engine MUST **not** optimize raw engagement. It MUST:
  (a) carry an explicit **objective = learning + sustained healthy engagement** in the re-ranker
  prompt and the candidate scoring (reward depth/variety/spacing, not session length); (b) enforce
  **diversity/novelty floors** (don't recommend the same kind/modality repeatedly; surface variety);
  (c) run **over-use detection** (from 0020 `days_active_28`/session signals) that, past a threshold,
  **de-escalates** (recommends a lighter load / suggests a rest) rather than pushing more; (d) never
  use **scarcity/fake-urgency/streak-pressure** framing in recommendation reasons (¬ß6.10).
- **FR-10 (transparency + control).** The system MUST expose a **"why am I seeing this"** reason per
  recommendation (the re-ranker's per-pick reason, sanitized) and a **reset** path
  (`POST /v1/me/recommendations/reset`) that clears the user's recommendation state/feedback and
  returns the user to cold-start defaults. iOS MUST surface both (a reason affordance on the activity
  card + a Settings "Personalization" control) (¬ß6.10).
- **FR-11 (evaluation harness).** An **offline eval harness** MUST exist: replay logged
  `(context ‚Üí recommendation ‚Üí outcome)` tuples from the lake and compute **NDCG\@k**,
  **precision/recall\@k** (relevance = the user engaged/completed/passed), **diversity** (intra-list
  modality/kind diversity), **novelty/coverage**, and **difficulty calibration** (predicted vs actual
  pass rate). Runs **offline in CI** against a fixture; against the real lake on demand (¬ß6.9).
- **FR-12 (mocked-re-ranker determinism).** The re-ranker call MUST be **monkeypatchable** (like
  `agent._runtime`); with a stubbed Bedrock response the full pipeline (candidates ‚Üí re-rank ‚Üí typed
  output) MUST be **deterministic and unit-testable**, and a **malformed/empty** LLM response MUST
  fall back to the **candidate prior order** (never crash, never empty) (¬ß6.3, ¬ß8).

### Non-functional

- **NFR-1 (latency).** Online `GET /v1/me/recommendations` and internal `recommend()` target
  **p50 ‚â§ 1.5 s, p95 ‚â§ 4 s** on a **cache miss** (one small Bedrock re-rank over a capped candidate
  set, `max_tokens` ‚â§ 800), and **p50 ‚â§ 50 ms** on a **cache hit** (DDB read). Candidate generation
  reads precomputed features/co-occurrence ‚Äî **no Athena/lake scan at request time**. The 0038
  Personalizer call rides inside that stage's budget (0038 NFR-1 hit path p50 ‚â§ 6 s).
- **NFR-2 (cost).** Per recommendation = **one small Bedrock call** (re-rank), amortized by caching
  (¬ß6.8) and by sharing the result across the Personalizer + the `GET` surface within a TTL. The heavy
  collaborative aggregates are **batch-precomputed** (rides the 0020 daily job ‚Äî no extra per-request
  model cost). Token/cost is logged (0027) and alarmed (0032). Recommendations are **free** (no
  credit, [`0023`]); only a true personalized re-generation in 0038 costs a credit.
- **NFR-3 (security/IAM).** Bedrock via **IAM only, no API key**. The recommendation Lambda gets
  **read** on `MangoFeatures-<stage>`, `bedrock:InvokeModel` on the **specific re-ranker model ARN**,
  **write** on the recommendation/feedback DDB items + the 0027 artifact prefix + 0015 events; **no**
  product-table write beyond its own items; **no** broad lake read at runtime (the batch job, not the
  online Lambda, reads the lake). User/book text in the re-rank prompt is **input-tagged untrusted**
  ([`0030`]). (Mirrors the least-privilege pattern in `api_stack.py`.)
- **NFR-4 (privacy).** Store only **non-sensitive aggregates/enums/ids** in recommendation + feedback
  items (no titles, free text, emails, precise location) ‚Äî same posture as [`0020`]/[`0015`].
  Per-user recommendation artifacts live under `users/<sub>/` so `DELETE /v1/me` purges them; the
  **feedback + recommendation DDB items are keyed `USER#<sub>`** and MUST be added to the
  `delete_account` purge (a small, tracked delta). Recommendation reasons shown to the user contain no
  other user's data.
- **NFR-5 (no DDB floats).** Scores, weights, difficulty, calibration MUST be **scaled ints (basis
  points 0‚Äì10000) or JSON strings**, never Python `float` ‚Äî through a single `shared/recommend.py`
  choke point (the `progress.py`/`features.py` rule).
- **NFR-6 (stdlib + boto3).** New Lambdas/modules use **stdlib + boto3 only** (no packaging). The
  offline eval harness may use stdlib math; **no** new packaged dependency. `cdk synth -c stage=beta`
  + `pytest` (moto + monkeypatched Bedrock) pass offline.
- **NFR-7 (rate-limit).** The recommendation endpoint sits behind the [`0029`] per-user/IP limiter +
  stage throttle so the Bedrock re-rank is not loop-callable (denial-of-wallet). Caching (¬ß6.8) is the
  first line of defense.
- **NFR-8 (style/contract).** black (100) + flake8 (120); `openapi.yaml` ‚áÑ `DTOs.swift` ‚áÑ handlers in
  lockstep; lenient DTO decode (unknown enum ‚Üí safe fallback) per the `CatalogBook.init(from:)` idiom.

## 6. Design

### 6.1 Architecture ‚Äî hybrid, three-layer

```mermaid
flowchart TD
    subgraph ONLINE [Online recommend() ‚Äî one small Bedrock call, cached]
      direction TB
      CTX["context: userSub, surface, trackId?,\nrecent feedback, time-of-day"]
      F["read MangoFeatures USER#sub\n(single-digit-ms, 0020)"]
      H["read recent history\n(0026 activity state / last-N graded)"]
      subgraph L1 [Layer 1 ‚Äî Candidate generation]
        CB["content-based\n(metadata match: topic/type/modality/difficulty)"]
        CF["collaborative\n(precomputed co-occurrence, batch)"]
        KB["knowledge-based\n(pedagogy + lifecycle rules)"]
      end
      CTX --> F --> L1
      H --> L1
      CB --> U["union + de-dup + prior score\n(capped, shuffled)"]
      CF --> U
      KB --> U
      U --> L2["Layer 2 ‚Äî gen-AI re-ranker (Bedrock Claude)\nreason over features+history+feedback+goals\n‚Üí ordered ids + per-pick reason + next/difficulty/modality"]
      L2 --> OUT["typed Recommendation(s)\n+ track-personalization result"]
      L2 -. malformed/empty .-> FB["fallback: prior order"] --> OUT
    end

    OUT --> CACHE[(DDB rec cache\nUSER#sub / REC#surface)]
    OUT --> P038["Layer 3 ‚Äî 0038 Personalizer\napplies overlay to cached base track"]
    OUT --> API["GET /v1/me/recommendations"]
    OUT -. transcript .-> S3[(0027 artifact store\nusers/sub/recs/recId/*)]
    OUT -. recommendation_served .-> LAKE[(0015 events lake)]
    OUTCOME["activity start/complete/skip/thumb"] -. recommendation_outcome .-> LAKE
    LAKE --> EVAL["offline eval harness\nNDCG/precision@k, diversity, calibration\n(+ online A/B once G12)"]
    EVAL -. tuning .-> L1
    BATCH["0020 daily batch\n+ collaborative co-occurrence aggregates"] --> CF
```

**The three layers, concretely.**
- **Layer 1 ‚Äî Candidate generation** (`shared/recommend/candidates.py`, pure functions). Produces a
  **small, high-recall** set (target ‚â§ 25 candidates) from the three sources (FR-1). It is the
  **cold-start floor**: content + knowledge alone produce a full set with zero collaborative data.
  Collaborative candidates come from **precomputed aggregates** (next paragraph), never a live scan.
- **Layer 2 ‚Äî Gen-AI re-ranker** (`shared/recommend/reranker.py` + a new prompt in `shared/prompts`).
  **High-precision** ordering of the Layer-1 set via **one Bedrock Claude call** that *reasons* about
  fit and returns ordered ids + reasons + the next-activity/difficulty/modality decision (FR-2). This
  is the **LLM-as-re-ranker** pattern (cheap candidates, expensive-but-small rank).
- **Layer 3 ‚Äî Track personalization** (the **0038 Personalizer** stage). Consumes Layer-2 output and
  writes the **overlay** over the cached base track (0038 ¬ß6.6). This spec **produces the input**;
  0038 **applies** it. This keeps the 0028 per-book cache hot (personalization is a thin overlay, not
  a re-generation ‚Äî 0038 D-4).

**Collaborative signal without a trained model (v1).** We do **not** train a matrix-factorization /
two-tower model in v1. Instead the **0020 daily batch** (extended by this spec) computes cheap
**co-occurrence aggregates** from the events lake and writes them to `MangoFeatures` as **book/global
entities** (e.g. `entityId=OBJECTIVE#<id>` / `BOOK#<id>`, `featureName=next_success_kind_*`):
"given profile-cluster C and just-completed objective X, the activity-kinds/difficulties that led to
**completion + pass** most often." Layer 1 reads these like any other feature (single-digit-ms). This
gives a **real collaborative signal** with **no model serving** and **no request-time lake scan** ‚Äî
matching the barbell guidance (collaborative ground truth for warm data; content/knowledge for
cold). A trained model or **Amazon Personalize** is the documented **upgrade** (¬ß10 D-2) once volume
justifies it.

**Where embeddings / vector search fit ‚Äî v2, not v1 (decision ¬ß10 D-1).** v1 ranks a **metadata +
collaborative** candidate set; the LLM supplies the semantic reasoning in-prompt. **v2** adds
**Amazon Titan Text Embeddings v2** over activity/objective/book text and stores them in **Amazon S3
Vectors** (cheap, 100‚Äì800 ms) ‚Äî used for **(a)** semantic **content-feed** retrieval ([`0042`]),
**(b)** **next-book** similarity over the [`0009`] catalog, and **(c)** **RAG-over-history** (retrieve
the user's semantically-similar *past successes* as few-shot context for the re-ranker). v2 upgrades
to **OpenSearch Serverless** only if latency/scale demand it. Embeddings are **never on the v1
critical path** and never a hard dependency (offline-first invariant preserved: the on-device Mock
path needs none of this).

### 6.2 Layer 1 ‚Äî candidate generation (signals ‚Üí candidates)

Each candidate is `{candidateId, kind, source[], priorScoreBp, meta{typeId, modality, difficulty,
objectiveRef, topic}}`. Sources:

| Source | Input | How a candidate is produced | Cold-start? |
|---|---|---|---|
| **content-based** | `pref_topics`, onboarding `interests`, `reading_pace_wpm`, per-modality success (¬ß6.5); the track's activities (0038 base) / catalog items (0009) | score by metadata match: topic overlap, `modality` the user succeeds at, `difficulty` near the user's tolerance, `estimatedMinutes` fits available time | **yes** (uses profile only) |
| **collaborative** | precomputed co-occurrence aggregates in `MangoFeatures` (`OBJECTIVE#‚Ä¶`/`BOOK#‚Ä¶` entities) keyed by a coarse **profile cluster** | pull the kinds/difficulties that most often led to completion+pass for similar users after the same objective | no (needs warm data; absent ‚áí skipped) |
| **knowledge-based** | the track's **lifecycle/scaffold** (0039 state, Bloom ladder), due **review** cards (0016), under-practiced Bloom levels | rules: next unfinished objective; a due `spacedReview`; cover a missing Bloom level; never exceed `maxAttempts`; honor `assigned/in_progress` | **yes** (deterministic) |

Output: union, de-dup by `candidateId`, keep the top **K (‚â§25)** by blended prior, **shuffle** (so the
LLM isn't anchored by our order ‚Äî ¬ß6.3), tag each with its sources. **Pure & deterministic** given
inputs (seedable shuffle for tests).

### 6.3 Layer 2 ‚Äî the gen-AI re-ranker (prompt + parse + safety)

A new `shared/prompts.rerank_system()` / `rerank_user(features, history, feedback, candidates,
objective)` builds a **strict-JSON** instruction. Invoked via the existing `agent._invoke(system,
user, max_tokens=800)` ‚Üí `extract_json` (monkeypatchable `_runtime`).

**Prompt contract (sketch).**
- **System:** "You are Mango's learning-path recommender. Order the given candidate activities to
  **maximize this learner's durable learning and healthy engagement** ‚Äî depth, variety, appropriate
  challenge, spaced reinforcement ‚Äî **not** time spent. You may only reorder/select from the provided
  candidate ids; never invent activities. Prefer variety over repetition. Respect explicit negative
  feedback. Output strict JSON."
- **User (compact):** `READER FEATURES {allow-listed}`, `RECENT HISTORY [{kind,modality,score}√óN]`,
  `EXPLICIT FEEDBACK [{target,signal}]`, `GOALS/TOPICS [‚Ä¶]`, `CANDIDATES [{id,kind,modality,
  difficulty,topic,sources}‚Ä¶]`, `OBJECTIVE {next objective + Bloom}`.
- **Output JSON:** `{ "orderedIds": ["‚Ä¶"], "next": {"id":"‚Ä¶","difficulty":3,"modality":"audio"},
  "preferredTypes":["‚Ä¶"], "preferredModalities":["‚Ä¶"], "difficultyTarget":3,
  "reasons": {"<id>":"short learner-facing reason"}, "notes":"optional" }`.

**Bias + reliability mitigations (from the LLM-ranker literature).**
- **Position/popularity bias:** shuffle candidate order before prompting (¬ß6.2); instruct explicitly
  *"do not prefer an item because it is popular or appears first."* Cap K so the set fits comfortably.
- **History-order perception:** present recent history **newest-first with explicit recency labels**
  (the literature notes LLMs mis-perceive interaction order).
- **Determinism for tests:** `extract_json` tolerant parse; on malformed/empty/‚ùò unparZeable output,
  **fall back to the Layer-1 prior order** and a rules-only `next` (FR-12) ‚Äî the engine **always**
  returns a valid result.
- **Safety:** book/user text in the prompt is **input-tagged untrusted** and run under the [`0030`]
  Guardrails posture (denied topics, injection defense); reasons surfaced to the user are sanitized
  (no raw model meta-text). The re-ranker has **no tools** and cannot act ‚Äî it only orders ids.

### 6.4 Layer 3 ‚Äî track-personalization output (what 0038 consumes)

`recommend(context)` returns (for `surface=track`):
```jsonc

  "recId": "rec_7af3",
  "surface": "track",
  "trackId": "trk_‚Ä¶",
  "preferredTypes": ["long_answer", "conversation"],   // 0039 type ids
  "preferredModalities": ["text", "audio"],            // 0039 modalities
```

**rules default** (prior order) is returned. This guarantees a **good first experience** with zero
data and **never** worse than today's generic track.

- **Explicit feedback** (immutable log + latest): `PK=USER#<sub>`, `SK=RECFEEDBACK#<ts>#<recId>` ‚Üí

- **Collaborative aggregates** (written by the batch, read by Layer 1): `PK=OBJECTIVE#<id>` /
  `BOOK#<id>` in **`MangoFeatures`** (not the product table) ‚Äî keeps the single-table product data

**S3 (0027).** Per-user recommendation transcript: `users/<sub>/recs/<recId>/{candidates.json,

### 6.8 Serving, caching, cost

- **Read path.** `recommend(ctx)`: (1) try the **DDB rec cache** `REC#<surface>` ‚Äî if fresh (within
  `ttl`), return it (**p50 ‚â§ 50 ms**, no Bedrock). (2) Miss ‚Üí read `MangoFeatures` + recent history ‚Üí
  Layer 1 candidates ‚Üí **one Bedrock re-rank** (`max_tokens ‚â§ 800`) ‚Üí write cache (`ttlSec` default
  **900 s** for `track`/`next_activity`; longer for `next_book`) ‚Üí log (0027) + emit
  `recommendation_served` (0015) best-effort ‚Üí return.
- **Precompute the heavy parts.** Collaborative co-occurrence aggregates are computed by the **0020
  daily batch** (no request-time cost). The cache means the **Personalizer + the `GET` surface share**
  one re-rank within the TTL. A **pre-warm** option (rides 0038's catalog pre-warm) can seed
  `next_book`/`track` recs for active users off-peak.
- **Cost.** One small Bedrock call per miss; cache hit rate keeps average cost low; token/cost logged
  (0027) and budget-alarmed (0032). **Free** to the user (no credit). Rate-limited (0029) so it can't
  be looped.
- **Float-free** at every write (`scoreBp`/ints/JSON only), via the `shared/recommend.py` choke point.

### 6.9 Feedback loop + evaluation

- **Log everything (FR-6).** Each rec writes a transcript to **0027** (candidates, LLM verdict +
  reasons, picks, model/tokens/latency, `recId`) and emits **`recommendation_served`** to **0015**;
  the corresponding **`recommendation_outcome`** is emitted when the recommended thing is
  started/completed/skipped/thumbed (the existing `exercise_graded`/`lesson_completed` carry the
  score; a tiny join key `recId` ties them). Best-effort; never fails the response.
- **Offline eval (FR-11).** `shared/recommend/eval.py` + `handlers/recommend_eval.py` (on-demand /
  scheduled) replays lake tuples and computes, per cohort: **NDCG\@k**, **precision/recall\@k**
  (relevance = engaged ‚àß passed), **MAP**; **intra-list diversity** (distinct modalities/kinds),
  **novelty/coverage** (fraction of the catalog/kinds ever recommended), and **difficulty
  calibration** (predicted `difficultyTarget` vs actual pass-rate ‚Äî are we matching challenge to
  skill?). CI runs it on a **fixture**; an ops command runs it on the real lake (Athena). Results land
  in the lake/0027 for trend tracking.
- **Online A/B ‚Äî gated on G12.** The architecture review flags **no experimentation framework
  (G12)**. This spec **defines what A/B needs** (a stable per-user **bucket** from a hash of `sub`, a
  variant tag stamped on `recommendation_served`, holdout = "rules-only / no LLM re-rank") and how to
  read it (compare guardrail + quality metrics across buckets), but **degrades to offline-only** until
  G12 lands. When it does, the first experiments are **LLM-re-rank vs rules-only** and **difficulty
  personalization on/off**, judged on **¬ß6.10 guardrails first, engagement second**.
- **Guardrail metrics (the objective ‚Äî ¬ß6.10).** Primary success = **reflection/application
  completion rate, D7/D30 retention, spaced-review adherence, % of recommendations the user finds
  helpful (thumbs-up rate)**; **counter-metrics that must not regress = notification-disable rate,
  over-use incidence, skip rate, and any rise in same-day session count beyond a healthy band** (a
  *rise* in raw sessions is treated as a **warning, not a win** ‚Äî `docs/GAMIFICATION.md` ¬ß5/¬ß6).

### 6.10 Ethics ‚Äî healthy engagement, transparency, control (honor `docs/GAMIFICATION.md`)

This is a **hard requirement**, not a nicety (FR-9/FR-10), and follows the doc's manifesto
("optimize for the user's goal, not time-on-app"; "make leaving easy and stopping graceful";
"protect over-users").

- **Objective = learning + durable healthy engagement, encoded everywhere.** The re-ranker prompt
  (¬ß6.3) and the candidate prior (¬ß6.2) reward **depth, variety, spacing, appropriate challenge** ‚Äî
  *not* session length or raw streak pressure. The eval treats **rising session count as a warning,
  not a win** (¬ß6.9). This is the "editorial values beyond clicks" pattern
  ([arXiv 2004.09980](https://arxiv.org/pdf/2004.09980)).
- **Diversity/novelty floors (FR-9b).** No recommending the same kind/modality on repeat; enforce a
  minimum intra-list variety so the user isn't funneled into one addictive loop (counter to the
  feedback-loop risk, [arXiv 2505.00054](https://arxiv.org/pdf/2505.00054)).
- **Over-use detection ‚Üí de-escalate (FR-9c).** When `days_active_28`/session signals indicate
  unhealthy attachment, the engine recommends a **lighter** load and the nudge surface can suggest a
  **rest** ("your habit is strong ‚Äî rest is part of learning"), never *more* ‚Äî the doc's "protect
  over-users" principle. It **never** escalates frequency to chase engagement.
- **No dark-pattern framing (FR-9d).** Recommendation `reason` strings are **honest and calm** ‚Äî no
  fake scarcity, fake urgency, fake-social, or "you'll lose everything" pressure. Reasons are
  sanitized and reviewed against a denylist of manipulative phrasings.
- **Transparency (FR-10).** Every recommendation carries a **"Why am I seeing this?"** reason (the
  re-ranker's per-pick reason, plain-language). The test for every reason mirrors the doc: *would we
  be comfortable showing the user exactly why this is here?*
- **Control (FR-10).** A **reset** (`POST ‚Ä¶/reset`) clears personalization ‚Üí cold-start; explicit
  **thumbs/skip/too-easy/too-hard** feedback is honored quickly; the user can keep using the app with
  **no personalization** at all (cold-start is a complete experience). Personalization is **opt-out
  respecting** the existing analytics opt-out ‚Äî if [`0015`] analytics are off, the engine runs on
  on-device context + onboarding only and **does not** read/write the lake.

### 6.11 Privacy & safety

- Store only **aggregates/enums/ids** (NFR-4); reasons contain no other user's data. Per-user
  artifacts under `users/<sub>/` (purged by `DELETE /v1/me`); **add** the new `REC#*`/`RECFEEDBACK#*`
  DDB items to the `delete_account` purge (tracked delta). Honor the analytics **opt-out** (¬ß6.10).
- Re-rank prompt text is **input-tagged untrusted** under [`0030`] (denied topics, injection
  defense). The re-ranker is **tool-less** and **action-less** (orders ids only). No PII in events
  (rides the 0015 typed/closed taxonomy).

## 10. Risks & open decisions

- **R-1 Cold-start (new users/items dominate early).** *Mitigation:* the **content + knowledge layer
  is the floor** and produces a full, sensible set with zero collaborative data (FR-1/FR-8); the LLM
  re-ranks on content alone; collaborative weight grows as warm data accrues (the barbell strategy).
  Cold-start is a **complete** experience, never a degraded one.
- **R-2 LLM-ranker popularity/position bias + history-order misperception.** *Mitigation:* shuffle
  candidates, cap K, explicit anti-popularity instruction, newest-first labeled history, and a
  **rules fallback** on any parse failure (¬ß6.3); track via eval (diversity/coverage) so funneling is
  caught.
- **R-3 Cost / denial-of-wallet (a Bedrock call per rec).** *Mitigation:* aggressive **caching**
  (shared across Personalizer + `GET`), **batch-precomputed** collaborative signals (no request-time
  model), small `max_tokens`, [`0029`] rate-limit, [`0032`] budget alarms; recs are **free** (no
  credit churn loop).
- **R-4 Ethical engagement / addictive feedback loop.** *Mitigation:* the entire ¬ß6.10 ‚Äî objective =
  learning+healthy engagement (not time), diversity floors, over-use **de-escalation**, no
  dark-pattern reasons, transparency + reset, and an **eval that treats rising sessions as a
  warning**. This is a first-class requirement (FR-9/FR-10), not an afterthought.
- **R-5 Popularity/feedback bias in collaborative aggregates.** *Mitigation:* novelty/coverage in the
  eval; cap the collaborative weight; the knowledge layer guarantees pedagogically-required items
  surface regardless of popularity.
- **R-6 Depends on 0020 features + 0026 history + 0038 Personalizer.** *Mitigation:* the spec is
  **consumer-shaped** and degrades ‚Äî cold-start when features are sparse, on-device `context` when
  0026 isn't ready, identity overlay when disabled; lands incrementally (¬ß9).
- **R-7 Eval validity (offline gains don't always transfer; no A/B yet ‚Äî G12).** *Mitigation:* offline
  metrics are **directional**; the real gate is the **guardrail metrics** once A/B (G12) lands; until
  then dark-launch + transcript review (0027) + conservative defaults.
- **Decisions needed (with recommendations):**
  - **D-1 (recommend: embeddings/vector search = v2, not v1).** v1 ranks a metadata+collaborative
    candidate set with an LLM re-ranker (no vector infra); add **Titan v2 + S3 Vectors** in v2 for
    semantic content-feed / next-book / RAG-over-history. Rationale: v1 needs no vector store to be
    good, keeps cost/latency/ops minimal, and preserves the offline-first invariant; embeddings buy
    semantic recall we don't yet need at our catalog size.
  - **D-2 (recommend: build on Bedrock + serverless primitives for v1; revisit Amazon Personalize
    later).** A managed recommender (Personalize) or a trained two-tower/MF model is the **upgrade**
    once interaction volume is high and warm ‚Äî the research shows CF wins on warm data. For v1, the
    LLM-re-rank-over-candidates approach needs **no training data**, reuses our `agent.py`/IAM path,
    and ships now. Keep a clean seam so a trained ranker can replace Layer 2's scoring later.
  - **D-3 (recommend: collaborative signal via precomputed co-occurrence, not a live model).** Compute
    cheap aggregates in the 0020 batch; read them like features. Avoids model serving + request-time
    lake scans; upgradeable to a real model under the same Layer-1 interface.
  - **D-4 (recommend: re-rank is a separate, cheaper model than the 0038 Planner).** Re-ranking a
    capped id set is a light task; use a fast/cheap `RERANK_MODEL_ID` to bound latency/cost (the
    Planner/Verifier can stay on Opus).
  - **D-5 (recommend: optimize healthy engagement, treat session-count as a counter-metric).** Lock
    the objective to learning + durable engagement; never ship an experiment won purely on
    time-on-app. (Non-negotiable per `docs/GAMIFICATION.md`.)
  - **D-6 (recommend: cache + share one re-rank across surfaces within TTL).** The Personalizer and the
    `GET` surface read the same cached result; refresh on TTL or explicit feedback.
